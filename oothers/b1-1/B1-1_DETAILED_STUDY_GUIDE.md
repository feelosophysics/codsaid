# B1-1 아주 상세한 학습 가이드

이 문서는 `mission.md`, `README.md`, `checklist/part1.png` ~ `checklist/part4.png`, 그리고 `scripts/` 디렉토리의 내용을 바탕으로 만든 학습 가이드입니다.

목표는 딱 두 가지입니다.

1. 미션을 실제로 수행할 때 무엇을 왜 하는지 이해한다.
2. 체크리스트 항목을 질문받았을 때 설명하고, 검증 명령어까지 자신 있게 말할 수 있다.

이 문서는 특히 다음 사람을 위해 썼습니다.

- `scripts` 폴더가 낯선 사람
- Bash 문법이 거의 처음인 사람
- "명령어는 따라 칠 수 있는데 뜻은 잘 모르겠다"는 사람
- 체크리스트에서 무엇을 물어보는지 감이 잘 안 오는 사람

## 0. 당신의 현재 환경에서 꼭 알아야 할 사실

사용자 환경 조건:

- Mac: **iMac Intel**
- macOS 호스트에서는 `sudo` 권한이 막혀 있음
- OrbStack은 설치되어 있음
- OrbStack 사용법은 아직 익숙하지 않음

이 조건에서 가장 중요한 결론은 다음입니다.

### 0-1. 이 미션은 macOS 호스트에서 하는 과제가 아니라, OrbStack 안의 Linux에서 하는 과제다

이 미션의 핵심 작업은 아래와 같습니다.

- `/etc/ssh/sshd_config` 수정
- `ufw` 활성화
- 사용자/그룹 생성
- `/var/log/agent-app` 생성 및 권한 변경
- `cron` 등록

이런 작업은 모두 **리눅스 시스템 권한**이 필요한 작업입니다.

즉, macOS에서 `sudo`가 막혀 있는 것은 크게 중요하지 않습니다.

정말 중요한 것은:

> OrbStack 안에 띄운 Ubuntu 환경에서 root 또는 sudo 권한을 쓸 수 있는가

입니다.

### 0-2. Intel Mac이라서 아키텍처 측면은 유리하다

이 워크스페이스의 `agent-app`은 **x86-64 Linux ELF** 실행 파일입니다.

사용자가 Intel Mac이므로, OrbStack에서 `amd64` 계열 Ubuntu를 쓰는 흐름이 자연스럽습니다.

즉, Apple Silicon처럼 "ARM이라 바이너리가 안 맞는 문제"는 상대적으로 덜 걱정해도 됩니다.

### 0-3. OrbStack에서 가장 먼저 해 볼 것

터미널에서:

```bash
orbctl status
```

이 명령으로 OrbStack이 켜져 있는지 확인합니다.

`Stopped`가 나오면:

1. OrbStack 앱을 먼저 실행
2. 다시 `orbctl status` 확인

### 0-4. OrbStack Linux 쉘에 들어가는 가장 쉬운 방법

OrbStack 도움말 기준으로 `orb`는 그냥 리눅스 쉘로 들어가는 명령입니다.

가장 쉬운 시작:

```bash
orb
```

특정 머신으로 들어가고 싶으면:

```bash
orb -m ubuntu-agent
```

루트 쉘로 바로 들어가고 싶으면:

```bash
orb -m ubuntu-agent -u root
```

이게 중요한 이유:

> 리눅스 안에서 `sudo`가 막혀 있더라도, OrbStack CLI로 root 사용자로 바로 진입할 수 있으면 미션 수행이 가능할 수 있습니다.

### 0-5. OrbStack 머신이 아직 없다면

OrbStack CLI 도움말 기준 생성 명령 형태는 아래입니다.

```bash
orb create -a amd64 ubuntu:22.04 ubuntu-agent
```

뜻:

- `-a amd64`: x86-64 아키텍처 사용
- `ubuntu:22.04`: Ubuntu 22.04 생성
- `ubuntu-agent`: 머신 이름

### 0-6. 이 미션은 어디서부터 어디까지를 "리눅스 안에서" 해야 하나?

아래 작업은 전부 OrbStack Ubuntu 안에서 해야 합니다.

- `apt install`
- `ufw`
- `useradd`, `groupadd`, `usermod`
- `mkdir /var/log/...`
- `chown`, `chmod`, `setfacl`
- `crontab`
- `./agent-app` 실행
- `monitor.sh` 실행

즉, macOS 터미널은 OrbStack으로 들어가는 "입구"이고, 실제 실습 무대는 OrbStack Linux입니다.

### 0-7. 파일은 어떻게 Linux로 옮기나?

OrbStack CLI 도움말 기준:

```bash
orbctl push -m ubuntu-agent agent-app
orbctl push -m ubuntu-agent scripts
```

또는 OrbStack 공유 폴더를 써도 됩니다.

중요한 생각:

- macOS의 현재 워크스페이스 파일
- OrbStack Linux 안에서 실제 실행할 파일

이 둘은 구분해서 생각해야 합니다.

### 0-8. 정말 막히는 경우: Linux 안에서도 root/sudo가 전혀 안 된다면?

이 경우는 솔직히 말해 **미션 자체가 성립하지 않습니다**.

왜냐하면 아래는 일반 사용자 권한만으로는 수행할 수 없기 때문입니다.

- SSH 설정 파일 수정
- 방화벽 활성화
- 시스템 사용자/그룹 생성
- `/var/log` 아래 디렉토리 관리
- cron 등록 환경 설계 일부

따라서 OrbStack Ubuntu 안에서도:

- `sudo` 가능하거나
- `root` 사용자로 진입 가능해야

합니다.

이 점은 과제 수행 전에 제일 먼저 확인해야 합니다.

### 0-9. 사용자의 현실에 맞는 가장 추천 경로

1. macOS에서 OrbStack 앱 실행
2. 터미널에서 `orbctl status` 확인
3. 필요하면 `orb create -a amd64 ubuntu:22.04 ubuntu-agent`
4. `orb -m ubuntu-agent -u root`로 리눅스 root 쉘 진입
5. 그 안에서 미션 수행
6. 필요하면 일반 사용자/운영 사용자로 다시 전환해 앱과 스크립트 실행 테스트

즉, 현재 사용자에게 가장 중요한 전략은:

> macOS 권한 문제를 해결하려고 애쓰기보다, OrbStack Linux 내부 권한을 먼저 확보하는 것

입니다.

## 1. 먼저 큰 그림부터 이해하기

이 미션은 단순히 "쉘 스크립트 하나 만드는 과제"가 아닙니다.

실제로는 아래 5개를 한 묶음으로 배우는 과제입니다.

1. 서버 기본 보안
2. 사용자/그룹/권한 설계
3. 애플리케이션 실행 환경 구성
4. 시스템 상태 점검 자동화
5. cron 기반 주기 실행과 로그 관리

즉, 이 미션은 다음 질문에 답할 수 있어야 끝난 것입니다.

- 누가 서버에 접속할 수 있는가?
- 어떤 포트만 외부에 열려 있는가?
- 누가 어떤 디렉토리를 읽고 쓸 수 있는가?
- 앱이 정상 실행 중인지 어떻게 확인하는가?
- 장애가 났을 때 어떤 기록을 보고 원인을 추적하는가?
- 로그가 너무 커졌을 때 어떻게 통제하는가?

## 2. 이 디렉토리 안의 파일들은 각각 뭐 하는 건가?

`b1-1` 디렉토리 안의 핵심 파일 역할은 아래와 같습니다.

### `mission.md`

과제의 원문입니다. 요구사항, 최종 산출물, 보너스 과제, 제약 사항이 적혀 있습니다.

### `README.md`

미션 요약판입니다. 핵심 요구사항이 다시 정리되어 있습니다.

### `checklist/part1.png` ~ `part4.png`

실제 평가/질문 포인트라고 보면 됩니다.

즉, 단순히 구현만 하는 것이 아니라 아래도 설명할 수 있어야 한다는 뜻입니다.

- 왜 그렇게 설정했는가
- 어떤 명령어로 확인했는가
- 왜 그 명령어를 골랐는가
- 운영 관점에서 어떤 문제가 생길 수 있는가

### `scripts/setup.sh`

전체 환경을 자동으로 깔아 주는 예시 스크립트입니다.

이 파일은 "미션 전체를 Bash로 어떻게 자동화할 수 있는가"를 보여줍니다.

### `scripts/monitor.sh`

미션의 핵심 스크립트입니다.

이 스크립트는 다음을 합니다.

- 앱 프로세스가 떠 있는지 확인
- 앱 포트가 LISTEN 상태인지 확인
- 방화벽 활성 여부 확인
- CPU/MEM/DISK 수치 수집
- 임계값 넘으면 경고 출력
- 로그 파일에 한 줄씩 누적 기록
- 로그가 너무 커지면 회전(log rotation)

### `scripts/report.sh`

보너스 과제용 요약 리포트 스크립트입니다.

`monitor.log`를 읽어 평균/최대/최소를 계산합니다.

### `scripts/archive.sh`

보너스 과제용 로그 보존 스크립트입니다.

- 오래된 로그 압축
- 아카이브 디렉토리로 이동
- 더 오래된 압축 파일 삭제

### `agent-app`

이 파일은 제공된 실행 대상 앱입니다.

중요한 점:

- 이 워크스페이스의 `agent-app`은 파이썬 소스 파일이 아니라 **Linux ELF 실행 파일**입니다.
- 즉, `python agent_app.py`처럼 읽어서 실행하는 스크립트가 아니라, 이미 빌드된 실행 파일입니다.
- 미션 문서에 `agent_app.py(또는 제공 앱 파일명)`이라고 적혀 있는데, 이 워크스페이스에서는 실제 제공 파일명이 `agent-app`입니다.
- 그래서 `monitor.sh`의 프로세스 검사 대상도 `agent-app`으로 잡아 둔 것입니다.

## 4. 체크리스트 Part 1 해설

Part 1은 "기본 구현이 실제로 되어 있는가"를 보는 영역입니다.

### 4-1. SSH 포트가 20022로 변경되었고 Root 원격 접속이 차단되었는가?

이 항목의 핵심은 두 가지입니다.

1. SSH 기본 포트를 22에서 20022로 바꿨는가
2. `root` 계정으로 직접 원격 로그인하지 못하게 했는가

#### 왜 필요한가?

- 기본 포트 `22`는 인터넷 상의 무차별 스캔 공격이 가장 많이 들어오는 포트입니다.
- 포트를 바꾸면 보안이 완성되는 것은 아니지만, 자동화된 저수준 공격을 크게 줄일 수 있습니다.
- `root`는 시스템 전체 권한을 가지므로, 원격에서 root 직접 로그인을 허용하면 피해 범위가 매우 커집니다.
- 그래서 실무에서는 일반 계정으로 로그인한 뒤 `sudo`로 필요한 작업만 수행하게 합니다.

#### 어디를 확인해야 하나?

보통 `/etc/ssh/sshd_config`를 봅니다.

확인 명령어 예시:

```bash
grep -E '^Port|^PermitRootLogin' /etc/ssh/sshd_config
```

이 명령어 뜻:

- `grep`: 텍스트에서 조건에 맞는 줄을 찾는 명령어
- `-E`: 정규표현식 확장 사용
- `^Port`: 줄의 시작이 `Port`인 줄
- `|`: 또는
- `^PermitRootLogin`: 줄의 시작이 `PermitRootLogin`인 줄

기대 결과 예시:

```bash
Port 20022
PermitRootLogin no
```

실제로 포트가 열렸는지도 봐야 합니다.

```bash
ss -tulnp | grep ssh
```

이 명령어 뜻:

- `ss`: 소켓 상태 조회
- `-t`: TCP
- `-u`: UDP
- `-l`: LISTEN 중인 소켓
- `-n`: 포트를 숫자로 표시
- `-p`: 프로세스 정보 표시

즉, "어떤 포트가 열려 있고 누가 잡고 있는지" 확인하는 명령어입니다.

체크리스트에서 이 항목을 설명할 때는 이렇게 말하면 좋습니다.

> `sshd_config`에서 설정값을 확인하고, `ss -tulnp`로 실제 `sshd`가 20022에서 리슨 중인지 검증했습니다. 설정 파일만 바뀌고 서비스 재시작이 안 됐을 수도 있기 때문에 둘 다 확인해야 합니다.

### 4-2. 방화벽이 활성화되어 있고 20022/tcp, 15034/tcp만 허용되는가?

#### 이 항목이 보는 것

- 방화벽이 켜져 있는가
- 허용 포트가 최소화되어 있는가
- SSH와 앱 포트만 열었는가

#### 왜 필요한가?

서버 보안의 기본 원칙은:

> 기본 차단, 필요한 것만 허용

입니다.

즉,

- SSH 관리용 포트 `20022`
- 앱 서비스 포트 `15034`

만 허용하고 나머지는 막아야 합니다.

#### UFW를 쓰는 이유

Ubuntu에서는 보통 `ufw`가 가장 이해하기 쉽습니다.

`ufw`는 `iptables`를 더 쉽게 다루게 해 주는 도구입니다.

#### 확인 명령어 예시

```bash
ufw status
```

또는 자세히:

```bash
ufw status verbose
```

기대 결과 예시:

```bash
Status: active

To                         Action      From
--                         ------      ----
20022/tcp                  ALLOW       Anywhere
15034/tcp                  ALLOW       Anywhere
```

설명할 때 핵심 포인트:

- 방화벽 "활성 여부"와 "허용 룰 목록"은 별개입니다.
- 단순히 `allow 20022/tcp`를 추가했다고 끝이 아니라 `Status: active`도 확인해야 합니다.

좋은 설명 예시:

> UFW를 사용했고, 기본 정책은 incoming deny, outgoing allow로 두었습니다. 그 위에 관리 포트 20022/tcp와 앱 포트 15034/tcp만 허용했습니다. 그래서 외부 노출 면적이 최소화됩니다.

### 4-3. `agent-admin/dev/test` 계정과 `agent-common/core` 그룹이 요구사항대로 구성되었는가?

#### 이 항목의 목적

서버를 혼자 쓰지 않고, 역할이 다른 여러 사람이 함께 쓰는 상황을 가정합니다.

계정:

- `agent-admin`: 운영/관리, cron 실행자
- `agent-dev`: 개발/운영, `monitor.sh` 작성자
- `agent-test`: QA/테스트

그룹:

- `agent-common`: 세 계정 모두 포함
- `agent-core`: admin, dev만 포함

#### 왜 그룹을 나누나?

이유는 "누구에게 어디까지 보여주고 어디까지 쓰게 할 것인가"를 역할별로 나누기 위해서입니다.

예를 들면:

- 업로드 디렉토리는 세 명 다 써도 됨
- 키 파일 디렉토리와 로그 디렉토리는 민감하므로 core만 접근

#### 확인 명령어

```bash
id agent-admin
id agent-dev
id agent-test
```

이 명령어는 사용자 UID, 기본 그룹, 추가 그룹을 보여줍니다.

기대 해석:

- `agent-admin`은 `agent-common`, `agent-core` 둘 다 있어야 함
- `agent-dev`도 둘 다 있어야 함
- `agent-test`는 `agent-common`만 있어야 함

설명 포인트:

> 사용자마다 권한을 개별적으로 붙이지 않고 그룹 기반으로 관리하면 확장성과 일관성이 좋아집니다. 사용자가 늘어날 때 그룹만 추가/제거하면 되기 때문입니다.

### 4-4. 앱이 Boot Sequence 5단계 [OK]를 통과하고 `Agent READY`가 출력되는가?

이 항목은 "환경이 제대로 구성되었는가"를 한 번에 검증합니다.

앱은 대충 실행만 되는 것이 아니라, 실행 전에 여러 조건을 확인합니다.

보통 이런 것들을 검사합니다.

1. 올바른 사용자 계정으로 실행 중인가
2. 필요한 환경 변수가 있는가
3. 필요한 파일이 존재하는가
4. 포트가 사용 가능한가 혹은 정상 상태인가
5. 로그 디렉토리에 쓸 수 있는가

#### 여기서 중요한 환경 변수

미션 문서 기준:

- `AGENT_HOME`
- `AGENT_PORT`
- `AGENT_UPLOAD_DIR`
- `AGENT_KEY_PATH`
- `AGENT_LOG_DIR`

#### 키 파일

다음 파일이 필요합니다.

```bash
$AGENT_HOME/api_keys/t_secret.key
```

파일 내용은 정확히 한 줄:

```bash
agent_api_key_test
```

#### 왜 환경 변수로 고정하나?

코드에 경로를 박아 두면 환경이 바뀔 때마다 수정이 필요합니다.

반면 환경 변수를 쓰면:

- 개발/테스트/운영 환경을 쉽게 나눌 수 있고
- 실행 계정에 따라 같은 앱을 다른 설정으로 띄울 수 있고
- cron이나 서비스 등록 시 같은 값을 재사용하기 좋습니다

설명 예시:

> 환경 변수는 실행 환경을 외부에서 주입하는 방식입니다. 코드 수정 없이 포트, 로그 경로, 키 경로를 바꿀 수 있고, cron 같은 비대화형 실행 환경에서도 동일한 설정을 재현하기 쉽습니다.

### 4-5. `monitor.sh`가 프로세스/포트 상태를 점검하고, 비정상 상태에서 `exit 1`로 종료되는가?

이 항목은 스크립트가 "장애를 감지하는가"를 봅니다.

`monitor.sh`는 좋은 의미의 "엄격한 감시자"여야 합니다.

즉,

- 앱이 안 떠 있으면 실패
- 포트가 안 열려 있으면 실패
- 이런 경우 성공한 척 로그만 남기면 안 됨

#### `exit 1`이 왜 중요한가?

유닉스 계열 시스템에서 종료 코드는 프로그램의 성공/실패를 나타냅니다.

- `exit 0`: 성공
- `exit 1` 이상: 실패

따라서 `cron`, 다른 스크립트, 운영자, 모니터링 도구가 "이 작업이 정상적으로 끝났는가?"를 기계적으로 판단할 수 있습니다.

#### 체크 시나리오

1. 앱 실행 중인 상태에서 `monitor.sh` 실행 -> 성공
2. 앱 종료 후 `monitor.sh` 실행 -> 프로세스 실패, `exit 1`
3. 포트가 안 열려 있는 상태 -> 포트 실패, `exit 1`

설명 예시:

> 프로세스와 포트는 서비스 생존 여부를 판단하는 최소 조건입니다. 둘 중 하나라도 실패하면 스크립트는 성공으로 간주하면 안 되기 때문에 `exit 1`로 종료해야 합니다.

### 4-6. `/var/log/agent-app/monitor.log`가 지정 포맷으로 누적 기록되는가?

이 항목이 보는 것은 로그 형식의 일관성입니다.

필수 포맷:

```bash
[YYYY-MM-DD HH:MM:SS] PID:... CPU:..% MEM:..% DISK_USED:..%
```

예시:

```bash
[2026-02-25 14:00:01] PID:48291 CPU:25.3% MEM:9.8% DISK_USED:23%
```

#### 왜 형식을 고정하나?

고정 포맷은 사람이 읽기 쉽고, 나중에 프로그램이 파싱하기도 쉽습니다.

`report.sh`가 통계를 낼 수 있는 것도 결국 로그 포맷이 일정하기 때문입니다.

#### "누적"의 의미

이건 매우 중요합니다.

로그는 기존 내용을 덮어쓰면 안 됩니다.

- `>` : 기존 파일 내용을 덮어씀
- `>>` : 파일 끝에 이어서 추가

그래서 체크리스트 Part 3에서 `>`와 `>>` 차이를 설명할 수 있어야 하냐고 물어봅니다.

좋은 설명 예시:

> 모니터링 로그는 시계열 데이터이기 때문에 과거 기록이 남아 있어야 합니다. 그래서 `>`가 아니라 `>>`를 사용해 매 실행마다 한 줄씩 이어서 기록합니다.

### 4-7. cron 매분 실행으로 `monitor.log`가 자동 증가하는가?

이 항목은 사람이 수동으로 실행한 게 아니라, 시스템이 자동으로 정해진 주기에 실행하고 있는지를 확인합니다.

#### 왜 cron을 쓰나?

모니터링은 반복 작업입니다.

운영자가 매분 손으로 돌릴 수는 없습니다.

그래서 예약 실행 도구인 `cron`을 사용합니다.

예시 크론 표현식:

```bash
* * * * * /home/agent-admin/agent-app/bin/monitor.sh
```

앞의 5칸은:

1. 분
2. 시
3. 일
4. 월
5. 요일

`* * * * *` 는 "매분"입니다.

#### 왜 `/etc/environment` 이야기가 나오나?

cron은 우리가 터미널에서 로그인해서 쓰는 쉘 환경과 다릅니다.

즉:

- `.bashrc`가 자동으로 안 읽힐 수도 있고
- 환경 변수가 없을 수도 있고
- `PATH`가 다를 수도 있습니다

그래서 `setup.sh`는 `/etc/environment`에도 AGENT 관련 환경 변수를 넣어 둡니다.

그리고 예시 crontab은 이렇게 적습니다.

```bash
* * * * * . /etc/environment; $AGENT_HOME/bin/monitor.sh >> /tmp/cron.log 2>&1
```

이 한 줄의 의미:

- `. /etc/environment`: 환경 변수 불러오기
- `$AGENT_HOME/bin/monitor.sh`: 모니터 스크립트 실행
- `>> /tmp/cron.log`: 표준출력을 누적 기록
- `2>&1`: 표준에러도 같은 곳에 합치기

이 부분은 면접처럼 자주 물어볼 만한 포인트입니다.

### 4-8. `monitor.log` 용량 관리(10MB/10개)가 설정되어 있고 동작을 설명할 수 있는가?

모니터링 로그는 시간이 갈수록 계속 쌓입니다.

그대로 두면 디스크를 잠식합니다.

그래서 보존 정책이 필요합니다.

미션 요구:

- `monitor.log`가 커지면
- 최대 10MB 기준
- 최대 10개 파일 유지

#### 현재 `scripts/monitor.sh`의 방식

이 스크립트는 `logrotate`를 쓰지 않고 **스크립트 내부 로직으로 직접 회전**합니다.

동작 방식:

1. 현재 `monitor.log` 크기 확인
2. 10MB 이상이면 회전 시작
3. `monitor.log.9` -> `monitor.log.10`
4. `monitor.log.8` -> `monitor.log.9`
5. ...
6. `monitor.log` -> `monitor.log.1`
7. 10개 초과한 가장 오래된 파일 삭제

이 과정을 흔히 "rotation"이라고 부릅니다.

설명 예시:

> 무한히 한 파일에 쓰면 디스크 고갈 위험이 있으므로 크기 기준으로 로그를 회전시켰습니다. 가장 최신 원본은 `monitor.log`, 직전 파일은 `monitor.log.1`처럼 관리하고, 최대 10개까지만 유지하도록 했습니다.

### 5-1. `monitor.sh`에서 프로세스 식별과 포트 확인에 사용한 명령과 선택 이유를 설명할 수 있는가?

현재 `scripts/monitor.sh`는 다음 명령어를 사용합니다.

#### 프로세스 확인: `pgrep -f`

```bash
APP_PID=$(pgrep -f "$APP_PROCESS" | head -1)
```

뜻:

- `pgrep`: 프로세스 목록에서 이름 조건에 맞는 PID 찾기
- `-f`: 프로세스 이름뿐 아니라 전체 명령행(command line)까지 포함해서 찾기
- `head -1`: 여러 개 나와도 첫 번째 PID 하나만 사용

왜 `pgrep`를 썼는가?

- `ps aux | grep ...`보다 목적이 명확합니다.
- `grep` 프로세스 자신이 같이 잡히는 문제를 피하기 쉽습니다.
- PID만 바로 얻기 좋습니다.

설명 예시:

> 프로세스 생존 여부를 확인하기 위해 `pgrep -f`를 사용했습니다. 전체 명령행 기준으로 매칭해서 제공 앱 파일명이 명령행에 포함되면 잡을 수 있고, 성공 시 PID를 바로 얻을 수 있어 후속 로그 기록에도 사용하기 편합니다.

주의점:

- 프로세스 이름이 너무 짧거나 애매하면 의도치 않은 다른 프로세스가 잡힐 수 있습니다.
- 그래서 운영 환경에서는 더 구체적인 패턴을 쓰거나 실행 파일 절대 경로 기준으로 검증하기도 합니다.

#### 포트 확인: `ss -tulnp`

```bash
PORT_LISTEN=$(ss -tulnp 2>/dev/null | grep ":${APP_PORT} " | grep -c LISTEN)
```

뜻:

- `ss -tulnp`: LISTEN 중인 TCP/UDP 포트와 프로세스 정보 조회
- `2>/dev/null`: 에러 메시지는 버림
- `grep ":15034 "`: 15034 포트 관련 줄 찾기
- `grep -c LISTEN`: `LISTEN` 문자열이 들어간 줄 개수 세기

왜 `ss`를 썼는가?

- 요즘 리눅스에서 `netstat`보다 권장되는 편입니다.
- 빠르고 현대적이며 기본 탑재되는 경우가 많습니다.

설명 예시:

> 프로세스만 살아 있어도 서비스가 포트를 열지 못한 상태일 수 있으므로, 별도로 `ss -tulnp`로 포트가 실제 LISTEN 상태인지 확인했습니다. 서비스 관점에서는 프로세스와 포트를 둘 다 보는 것이 안전합니다.

### 5-2. CPU/MEM/DISK 값을 어떤 방식으로 추출했고, 로그 포맷을 왜 고정했는지 설명할 수 있는가?

#### CPU 사용률

현재 스크립트:

```bash
CPU_IDLE=$(top -bn1 | grep "Cpu(s)" | awk '{print $8}' | tr -d '%')
CPU_USAGE=$(awk "BEGIN {printf \"%.1f\", 100 - ${CPU_IDLE:-0}}")
```

아이디어:

- `top -bn1`: 배치 모드로 한 번만 출력
- `Cpu(s)` 줄에서 idle 비율 추출
- `100 - idle` 로 사용률 계산

왜 이런가?

`top`은 CPU의 각 상태 비율을 보여주는데, idle은 "쉬고 있는 비율"입니다.

따라서:

```text
CPU 사용률 = 100 - idle
```

주의:

- `top` 출력 형식은 배포판/언어/버전에 따라 다를 수 있어 파싱이 깨질 수 있습니다.
- 그래서 스크립트는 fallback용 `sed`도 하나 더 넣어 두었습니다.

#### 메모리 사용률

```bash
MEM_INFO=$(free | grep Mem)
MEM_TOTAL=$(echo "$MEM_INFO" | awk '{print $2}')
MEM_USED=$(echo "$MEM_INFO"  | awk '{print $3}')
MEM_USAGE=$(awk "BEGIN {printf \"%.1f\", ($MEM_USED / $MEM_TOTAL) * 100}")
```

뜻:

- `free`: 메모리 정보 출력
- `grep Mem`: 메모리 총계 라인만 선택
- 전체 메모리와 사용 메모리를 가져와 백분율 계산

#### 디스크 사용률

```bash
DISK_USED=$(df / | tail -1 | awk '{print $5}' | tr -d '%')
```

뜻:

- `df /`: 루트 파티션 `/`의 디스크 사용량 확인
- `tail -1`: 실제 데이터 줄 선택
- `$5`: 사용률 컬럼
- `%` 제거

#### 로그 포맷을 왜 고정하나?

로그를 사람이 읽는 것도 중요하지만, 나중에 프로그램이 재가공하기 쉬워야 합니다.

예를 들어 `report.sh`는 다음 전제를 깔고 있습니다.

- 날짜는 `[]` 안에 들어 있다
- `PID:`, `CPU:`, `MEM:`, `DISK_USED:` 같은 키가 고정이다
- 값 뒤에 `%`가 붙는다

그래서 형식이 흔들리면 통계 스크립트가 깨집니다.

설명 예시:

> 로그 형식을 고정한 이유는 후속 자동 분석을 위해서입니다. 사람이 눈으로 보기 쉽게 하면서도 `grep`, `awk`, 정규표현식으로 안정적으로 파싱할 수 있는 형태여야 했습니다.

### 5-3. 소유자(`agent-dev`)와 실행자(`agent-admin`, cron) 권한 정책을 어떻게 만족시켰는가?

이 항목은 아주 중요합니다.

미션 요구:

- 파일 소유자: `agent-dev`
- 그룹: `agent-core`
- 권한: `750`
- cron 실행자: `agent-admin`

#### `750` 권한 해석

숫자 권한은 아래 뜻입니다.

- `7` = `rwx`
- `5` = `r-x`
- `0` = `---`

즉:

- 소유자(`agent-dev`)는 읽기/쓰기/실행 가능
- 그룹(`agent-core`)은 읽기/실행 가능
- 기타 사용자는 아무 권한 없음

#### 왜 이 구성이 성립하나?

`agent-admin`은 파일 소유자는 아니지만 `agent-core` 그룹에 속해 있습니다.

따라서 그룹 권한 `r-x`로 스크립트를 실행할 수 있습니다.

즉:

- 작성/수정 책임은 `agent-dev`
- 운영 실행 책임은 `agent-admin`

이렇게 역할이 분리됩니다.

설명 예시:

> `monitor.sh`의 수정 권한은 개발 역할인 `agent-dev`가 가지고, 실제 운영 실행은 `agent-admin`이 담당하도록 분리했습니다. 이를 위해 파일 소유자는 `agent-dev`, 그룹은 `agent-core`, 권한은 `750`으로 두었고, `agent-admin`을 `agent-core`에 포함시켜 실행 가능하게 했습니다.

### 5-4. 용량 기반 로그 관리(10MB/10개)를 어떤 방식으로 구현했는가?

현재 구현은 스크립트 내부 함수 `rotate_log()`입니다.

핵심 코드 흐름:

1. `LOG_FILE`이 존재하는지 확인
2. `stat -c%s`로 파일 크기(byte) 확인
3. 최대 바이트와 비교
4. 기준 초과 시 번호가 큰 백업부터 뒤로 밀기
5. 현재 `monitor.log`를 `.1`로 바꾸기
6. 초과 개수 삭제

#### 왜 번호를 큰 것부터 이동하나?

만약 작은 번호부터 이동하면 덮어쓰기 충돌이 납니다.

예:

- `monitor.log.1`이 이미 있을 때
- 먼저 `monitor.log -> monitor.log.1`을 해 버리면
- 기존 `.1`이 날아갑니다

그래서 보통:

- `.9 -> .10`
- `.8 -> .9`
- ...

처럼 뒤에서부터 이동합니다.

설명 예시:

> 스크립트 내부에서 수동 rotation을 구현했습니다. 현재 로그가 10MB 이상이면 백업 파일을 뒤에서부터 한 칸씩 밀고, 원본을 `.1`로 이동한 뒤 새 로그를 계속 기록합니다. 이렇게 하면 파일 수를 10개로 제한하면서 최근 로그를 보존할 수 있습니다.

### 6-1. SSH 포트 변경과 Root 접속 차단이 왜 보안에 효과적인가?

이 질문은 단순 암기가 아니라 **위협 모델(threat model)** 관점 설명을 원합니다.

#### SSH 포트 변경

보안 수준을 마법처럼 올려 주는 것은 아닙니다.

하지만 자동화 스캔 봇은 주로:

- 22번 포트를 먼저 뒤지고
- 기본 계정명/root 계정을 대상으로
- 무차별 로그인 시도를 합니다

포트를 바꾸면:

- 노이즈가 줄고
- 불필요한 접근 로그가 줄고
- 아주 기초적인 자동 공격 노출이 감소합니다

즉, "강한 인증을 대체하는 보안"이 아니라 "노출면을 줄이는 보조 수단"입니다.

#### Root 로그인 차단

이건 훨씬 본질적인 보안입니다.

이유:

- root는 모든 권한을 가짐
- 탈취당하면 피해 범위가 최대
- 누가 어떤 작업을 했는지 사용자 단위 추적이 어려워짐

일반 계정 + `sudo` 방식을 쓰면:

- 개별 계정별 감사 추적이 가능하고
- 필요한 순간에만 권한 상승하고
- 원격 직접 root 노출을 막을 수 있습니다

짧은 모범 답변:

> SSH 포트 변경은 자동화된 저수준 스캔/무차별 대입 시도를 줄이는 데 도움이 되고, Root 원격 로그인 차단은 최고 권한 계정의 직접 노출을 막는 핵심 조치입니다. 즉 하나는 노출면 감소, 하나는 권한 남용 위험 감소에 해당합니다.

---

### 6-2. 왜 `api_keys`와 로그 디렉토리를 `agent-core`로 제한했는가?

이건 **최소 권한 원칙(Principle of Least Privilege)** 설명 문제입니다.

#### `api_keys`

여기는 비밀키가 있는 곳입니다.

따라서:

- 테스트 계정까지 접근할 이유가 없음
- 개발/운영 핵심 인력만 접근해야 함

#### 로그 디렉토리

로그는 겉보기엔 단순 기록 같지만, 실제로는 민감할 수 있습니다.

예:

- 내부 경로
- 프로세스 정보
- 장애 패턴
- 자원 사용 추세

따라서 모두에게 쓰기 권한을 주면:

- 로그 위조
- 장애 흔적 훼손
- 불필요한 정보 노출

문제가 생길 수 있습니다.

짧은 모범 답변:

> `api_keys`와 로그 디렉토리는 민감 정보와 운영 증적을 담고 있으므로 모든 사용자에게 열면 안 됩니다. 업무상 필요한 `agent-core`만 접근시키는 것이 최소 권한 원칙에 맞고, 보안과 감사 추적 측면에서도 적절합니다.

---

### 6-3. 왜 "경고는 출력하지만 종료하지 않는 항목"을 분리했는가?

현재 `monitor.sh`는 항목을 둘로 나눕니다.

#### 실패 시 즉시 종료해야 하는 것

- 프로세스 없음
- 포트 LISTEN 아님

이 둘은 서비스 생존성과 직결됩니다.

#### 경고만 출력하는 것

- 방화벽 비활성
- CPU 임계값 초과
- 메모리 임계값 초과
- 디스크 임계값 초과

이것들은 "문제의 징후"이긴 하지만, 그 순간 바로 서비스가 죽었다고 단정할 수는 없습니다.

예:

- CPU 25%는 경고지만 서비스는 정상일 수 있음
- 방화벽이 순간적으로 비활성이라도 앱 프로세스와 포트는 살아 있을 수 있음

운영 관점에서 이 분리가 중요한 이유:

- 치명 장애와 주의 신호를 구분할 수 있음
- 자동화가 과도하게 실패 처리되는 것을 막을 수 있음
- 후속 대응 우선순위를 나눌 수 있음

짧은 모범 답변:

> 서비스의 생존 여부를 판단하는 핵심 체크와, 운영상 주의가 필요한 상태를 분리했습니다. 프로세스/포트는 실패 시 즉시 종료해야 하지만, 방화벽 비활성이나 임계값 초과는 경고를 남기고 운영자가 추적할 수 있게 하는 편이 실무적으로 더 유용합니다.

---

### 6-4. `>` 와 `>>` 차이, 왜 로그 누적에 `>>`가 필요한가?

이 질문은 Bash 기초를 보는 매우 전형적인 질문입니다.

#### `>`

표준출력을 파일로 보냅니다.

그런데 기존 내용이 있으면 **덮어씁니다**.

예:

```bash
echo hello > a.txt
echo world > a.txt
```

최종 파일 내용:

```bash
world
```

#### `>>`

표준출력을 파일 끝에 **추가**합니다.

예:

```bash
echo hello >> a.txt
echo world >> a.txt
```

최종 파일 내용:

```bash
hello
world
```

#### 모니터링 로그에서 왜 `>>`가 필요한가?

모니터링 로그는 시계열 이력입니다.

과거 값이 계속 쌓여야:

- 추세를 볼 수 있고
- 평균/최대/최소를 계산할 수 있고
- 장애 직전 상태를 추적할 수 있습니다

그래서 `monitor.sh`는 로그 기록에 `>>`를 써야 합니다.

---

## 7. 체크리스트 Part 4 해설

Part 4는 "응용력과 운영 사고"를 봅니다.

---

### 7-1. 모니터링 대상이 웹 서버(Nginx 등)로 바뀌면 무엇을 바꿔야 하나?

핵심은 4가지입니다.

1. 프로세스 이름
2. 포트
3. 로그 경로/형식
4. 임계값 기준

예를 들어 Nginx라면:

- 프로세스 검사 대상: `nginx`
- 포트: `80`, `443` 또는 내부 프록시 포트
- 로그: `/var/log/nginx/access.log`, `/var/log/nginx/error.log`
- 임계값: 트래픽 특성에 맞게 재설정

현재 `monitor.sh`에서 바꿀 만한 변수:

```bash
APP_PROCESS="agent-app"
APP_PORT=15034
LOG_DIR="${AGENT_LOG_DIR:-/var/log/agent-app}"
CPU_THRESHOLD=20
MEM_THRESHOLD=10
DISK_THRESHOLD=80
```

즉, 미션의 핵심은 "정답 포트 15034를 외우는 것"이 아니라:

> 감시 대상 서비스가 바뀌면 무엇을 기준으로 모니터링 정의를 다시 세워야 하는지 이해하는 것

입니다.

---

### 7-2. 프로세스는 살아 있는데 포트가 안 열리는 상황을 발견했다면?

이건 실제 운영에서 자주 나오는 문제입니다.

가능한 원인 후보:

1. 앱이 시작은 했지만 포트 bind 단계에서 실패
2. 다른 프로세스가 이미 그 포트를 사용 중
3. 환경 변수 값이 잘못되어 다른 포트로 떴음
4. 앱이 내부 초기화 실패 후 반쯤 걸린 상태
5. 권한 또는 네트워크 설정 문제
6. 앱이 localhost에만 바인딩했고 `0.0.0.0`에 안 뜸

확인 순서 예시:

1. 프로세스 상세 확인

```bash
ps -ef | grep agent-app
```

2. 포트 상태 확인

```bash
ss -tulnp | grep 15034
```

3. 앱 실행 로그/콘솔 메시지 확인

4. 환경 변수 확인

```bash
sudo -u agent-admin bash -lc 'env | grep AGENT'
```

5. 포트 충돌 확인

```bash
ss -tulnp | grep -E ':15034 '
```

설명 예시:

> 프로세스 생존과 포트 개방은 별개입니다. 우선 실제로 어떤 포트에 바인딩됐는지 확인하고, 환경 변수 오설정인지 포트 충돌인지 앱 초기화 실패인지 순서대로 좁혀 가야 합니다.

---

### 7-3. 로그가 급증해 디스크가 가득 찰 위험이 있다면 운영자는 무엇을 해야 하나?

체크리스트는 단기 대응과 중기 대응을 모두 생각하는지 보려는 것입니다.

#### 단기 대응

- 로그 로테이션 즉시 점검
- 디스크 사용량 확인
- 비정상적으로 큰 로그 파일 확인
- 필요하면 오래된 로그 압축/이동
- 서비스 폭주나 오류 반복 여부 확인

예:

```bash
df -h /
du -sh /var/log/agent-app
ls -lh /var/log/agent-app
tail -n 100 /var/log/agent-app/monitor.log
```

#### 중기 대응

- 로그 보존 정책 강화
- 로그 레벨 조정
- 중앙 로그 수집 시스템 도입 검토
- 디스크 알람 추가
- 임계값과 주기 재조정

모범 답변:

> 단기적으로는 현재 디스크 사용량과 어떤 로그가 급증하는지 확인하고, 로테이션/압축/아카이브가 정상 동작하는지 점검해야 합니다. 중기적으로는 로그 보존 정책, 로그 레벨, 중앙 수집 체계, 디스크 알람 기준을 재설계해야 합니다.

## 8. `scripts/monitor.sh`를 정말 처음 보는 사람 기준으로 해설

이제 스크립트를 읽는 법 자체를 설명하겠습니다.

아래 설명은 현재 `scripts/monitor.sh`를 기준으로 합니다.

---

### 8-1. 첫 줄 `#!/bin/bash`

이 줄을 **shebang**이라고 부릅니다.

뜻:

> 이 파일을 실행할 때 `bash`로 해석해서 실행해라

입니다.

즉, 운영체제에게 "이건 Bash 스크립트야"라고 알려 주는 표시입니다.

---

### 8-2. 주석은 `#`

예:

```bash
# 감시할 프로세스명
APP_PROCESS="agent-app"
```

`#` 뒤는 실행되지 않고 사람이 읽는 설명입니다.

---

### 8-3. 변수 대입

```bash
APP_PROCESS="agent-app"
APP_PORT=15034
```

쉘 변수는 `=` 양옆에 공백을 두지 않습니다.

좋은 예:

```bash
APP_PORT=15034
```

나쁜 예:

```bash
APP_PORT = 15034
```

이건 Bash에서 오류입니다.

---

### 8-4. `${AGENT_LOG_DIR:-/var/log/agent-app}` 뜻

```bash
LOG_DIR="${AGENT_LOG_DIR:-/var/log/agent-app}"
```

이건 Bash의 기본값 문법입니다.

뜻:

- `AGENT_LOG_DIR` 환경 변수가 있으면 그 값을 쓰고
- 없으면 `/var/log/agent-app`를 쓰겠다

이런 패턴은 매우 자주 나옵니다.

장점:

- 환경 변수를 주입할 수 있는 곳에서는 유연하게 바꿀 수 있고
- 없을 때도 안전한 기본값으로 동작합니다

---

### 8-5. 함수 정의

예:

```bash
timestamp() {
    date "+%Y-%m-%d %H:%M:%S"
}
```

함수는 "자주 쓸 작업을 이름 붙여 묶어 둔 것"입니다.

`timestamp` 함수를 호출하면 매번 같은 형식의 시간을 얻습니다.

다른 함수:

```bash
rotate_log() {
    ...
}
```

이건 로그 회전 전용 함수입니다.

---

### 8-6. 명령 치환 `$(...)`

예:

```bash
APP_PID=$(pgrep -f "$APP_PROCESS" | head -1)
```

뜻:

- 괄호 안 명령을 실행한 결과를
- 문자열로 받아
- 변수에 넣는다

즉, `pgrep` 결과 PID를 `APP_PID` 변수에 담는 것입니다.

---

### 8-7. 조건문 `if [ ... ]; then ... fi`

예:

```bash
if [ -z "$APP_PID" ]; then
    ...
fi
```

여기서:

- `[ ... ]` 는 테스트 문법
- `-z` 는 문자열 길이가 0이면 참

즉:

> `APP_PID`가 비어 있으면

이라는 뜻입니다.

비슷한 테스트:

- `[ -f "$LOG_FILE" ]` : 일반 파일이면 참
- `[ -d "$LOG_DIR" ]` : 디렉토리면 참
- `[ "$x" -eq 0 ]` : 숫자가 0과 같으면 참
- `[ "$x" -gt 10 ]` : 숫자가 10보다 크면 참

---

### 8-8. 표준출력과 표준에러

리눅스 명령은 보통 두 종류의 출력을 냅니다.

- 표준출력(stdout)
- 표준에러(stderr)

예:

```bash
ss -tulnp 2>/dev/null
```

여기서 `2>`는 표준에러를 다른 곳으로 보내라는 뜻입니다.

`/dev/null`은 "버리는 통" 같은 특수 파일입니다.

즉:

> 에러 메시지는 화면에 보이지 않게 버린다

입니다.

왜 쓰나?

- 권한 경고나 불필요한 메시지로 파싱이 망가지지 않게 하려고
- 사용자에게 핵심 결과만 보여 주려고

---

### 8-9. 파이프 `|`

예:

```bash
ss -tulnp | grep ":${APP_PORT} " | grep -c LISTEN
```

파이프는 앞 명령의 출력을 뒤 명령의 입력으로 넘깁니다.

흐름:

1. `ss -tulnp`가 전체 포트 목록 출력
2. `grep`이 그중 15034 관련 줄만 선택
3. 또 다른 `grep -c LISTEN`이 LISTEN 줄 수만 셈

즉 "작은 도구들을 연결해 큰 작업 만들기"가 쉘의 핵심 스타일입니다.

---

### 8-10. `local` 변수

예:

```bash
local size_bytes
```

이건 함수 안에서만 쓰는 지역 변수입니다.

장점:

- 다른 변수와 이름 충돌을 줄임
- 함수가 바깥 상태를 함부로 오염시키지 않음

---

### 8-11. 로그 회전 함수 `rotate_log()` 읽는 법

핵심 흐름만 다시 정리하면:

```bash
if [ ! -f "$LOG_FILE" ]; then
    return
fi
```

뜻:

- 로그 파일이 아직 없으면
- 회전할 필요가 없으니 함수 종료

```bash
size_bytes=$(stat -c%s "$LOG_FILE" 2>/dev/null || echo 0)
```

뜻:

- 파일 크기를 바이트 단위로 구함
- 실패하면 `0` 사용

여기서 `||` 뜻:

> 앞 명령 실패 시 뒤 명령 실행

```bash
for i in $(seq $((MAX_LOG_FILES - 1)) -1 1); do
    ...
done
```

뜻:

- `MAX_LOG_FILES - 1` 부터 1까지 거꾸로 반복

왜 거꾸로?

앞에서 설명했듯 덮어쓰기 충돌을 막기 위해서입니다.

---

### 8-12. Health Check 블록 읽는 법

프로세스 검사:

```bash
APP_PID=$(pgrep -f "$APP_PROCESS" | head -1)
if [ -z "$APP_PID" ]; then
    echo "Checking process '$APP_PROCESS'... [FAIL] (Process not running)"
    echo "[$(timestamp)] HEALTH CHECK FAILED: process '$APP_PROCESS' not found" >> "$LOG_FILE"
    exit 1
fi
```

핵심 의미:

- PID를 찾는다
- 못 찾으면 실패 메시지 출력
- 실패 로그 기록
- 종료 코드 1로 스크립트 종료

포트 검사도 같은 구조입니다.

이 두 블록이 `monitor.sh`의 생명선입니다.

---

### 8-13. 방화벽 점검 블록 읽는 법

```bash
FIREWALL_OK=false
```

처음에는 "방화벽 괜찮지 않다"라고 가정합니다.

그 다음:

- `ufw` 명령이 있는지 본다
- 있으면 활성인지 본다
- 아니면 `firewall-cmd`도 시도한다

둘 다 아니면:

```bash
echo "[WARNING] Firewall is not active! System may be exposed."
```

즉, 경고만 찍고 계속 진행합니다.

이 블록이 바로 "경고는 출력하되 종료하지 않는 항목"의 예시입니다.

---

### 8-14. CPU/MEM/DISK 수집 블록 읽는 법

이 영역은 전부 "명령 결과를 읽어서 숫자만 뽑아내는" 작업입니다.

즉 Bash 스크립트는:

- 시스템 명령 실행
- 필요한 줄 선택
- 필요한 칸 선택
- 기호 제거
- 계산

을 반복합니다.

예를 들면:

```bash
df / | tail -1 | awk '{print $5}' | tr -d '%'
```

흐름:

1. `df /`: `/` 디스크 정보 출력
2. `tail -1`: 실제 데이터 줄 선택
3. `awk '{print $5}'`: 5번째 컬럼 선택
4. `tr -d '%'`: `%` 제거

이런 식으로 "문자열 파이프라인"을 만든다고 생각하면 됩니다.

---

### 8-15. 임계값 경고 블록

예:

```bash
CPU_INT=$(printf "%.0f" "$CPU_USAGE")
if [ "$CPU_INT" -gt "$CPU_THRESHOLD" ]; then
    echo "[WARNING] CPU threshold exceeded (${CPU_USAGE}% > ${CPU_THRESHOLD}%)"
fi
```

왜 정수로 바꾸나?

`[ ... -gt ... ]` 같은 Bash 숫자 비교는 보통 정수를 다루기 때문입니다.

`CPU_USAGE`는 `25.3` 같은 소수 문자열이라서, 먼저 반올림/정수화한 뒤 비교하는 것입니다.

---

### 8-16. 로그 기록 블록

```bash
LOG_LINE="[$(timestamp)] PID:${APP_PID} CPU:${CPU_USAGE}% MEM:${MEM_USAGE}% DISK_USED:${DISK_USED}%"
echo "$LOG_LINE" >> "$LOG_FILE"
```

이 블록은 지금까지 모은 핵심 정보를 한 줄의 문자열로 묶어 파일 끝에 붙입니다.

이것이 이 미션의 최종 증적 데이터입니다.

즉:

2026-02-25 13:00:00 < 2026-02-25 14:00:00

를 문자열 비교로 처리해도 시간 순서와 일치합니다.

입니다.

즉, 방어적인 코딩입니다.

## 10. `scripts/archive.sh`는 무슨 뜻인가?

핵심은 로그 수명 주기 관리입니다.

1. 일정 기간 지나면 압축
2. 따로 모아 보관
3. 더 오래되면 삭제

---

### 10-1. 왜 압축/이동/삭제가 필요하나?

로그는 운영 증적이라 너무 빨리 지우면 안 됩니다.

하지만 영원히 원본으로 쌓아 두면:

- 디스크 공간 낭비
- 파일 관리 어려움
- 검색 성능 저하

가 생깁니다.

그래서 보통:

- 최근 로그: 원본 유지
- 조금 지난 로그: 압축
- 오래된 로그: 아카이브
- 너무 오래된 로그: 삭제

같은 정책을 둡니다.

---

### 10-2. `find ... -mtime` 뜻

예:

```bash
find "$LOG_DIR" -maxdepth 1 -name "*.log" -mtime "+7"
```

뜻:

- `LOG_DIR` 아래
- 한 단계만 내려가서
- `*.log` 파일 중
- 수정 시각이 7일보다 오래된 것 찾기

운영 자동화에서 `find`는 거의 필수 도구입니다.

---

### 10-3. 왜 예외 처리를 넣었나?

스크립트는 현실에서 늘 예쁜 상황만 만나지 않습니다.

예:

- 디렉토리가 없음
- 권한 없음
- 대상 파일이 0개

이때 무조건 실패해 버리면 운영 자동화가 거칠어집니다.

그래서 `archive.sh`는:

- 디렉토리가 없으면 경고
- 권한 없으면 경고/실패 메시지
- 대상 없으면 건너뜀

같이 방어적으로 처리합니다.

즉, 미션 문서 전체를 거의 스크립트화한 것입니다.

### 11-2. 왜 `.bashrc`와 `/etc/environment`를 둘 다 건드리나?

#### `.bashrc`

- 주로 사용자가 로그인 쉘을 열었을 때 적용
- 대화형 작업에 유리

#### `/etc/environment`

- 시스템 전역 환경 변수 성격

즉:

- 사람이 로그인해서 앱 실행할 때도 값이 필요하고
- cron이 백그라운드에서 스크립트 실행할 때도 값이 필요하므로

둘 다 다룹니다.

---

### 11-3. 왜 `setfacl -d`를 같이 쓰나?

예:

```bash
sudo setfacl -d -m g:agent-common:rwx $AGENT_HOME/upload_files
```

여기서 `-d`는 **default ACL**입니다.

뜻:

> 이 디렉토리 안에 앞으로 새로 만들어질 파일/디렉토리에도 기본적으로 이 ACL을 물려줘라

입니다.

왜 중요하나?

디렉토리 자체만 권한을 맞추고 끝내면, 나중에 그 안에 새로 만든 파일이 다른 권한으로 생길 수 있습니다.

default ACL을 주면 협업 디렉토리 정책이 더 안정적으로 유지됩니다.

## 12. 꼭 알아야 하는 리눅스 권한 개념

이 미션을 이해하려면 아래 개념을 꼭 알아야 합니다.

---

### 12-1. 소유자(owner), 그룹(group), 기타(other)

리눅스 파일 권한은 기본적으로 세 층입니다.

1. 소유자
2. 그룹
3. 기타

예:

```bash
-rwxr-x---
```

해석:

- 소유자: `rwx`
- 그룹: `r-x`
- 기타: `---`

---

### 12-2. `chmod 750`, `chmod 2770`의 차이

`750`:

- 소유자 `rwx`
- 그룹 `r-x`
- 기타 `---`

`2770`:

- 앞의 `2`는 **setgid bit**
- 나머지 `770`은 권한

즉:

- 디렉토리에서 setgid가 걸려 있으면
- 그 안에 새로 생성되는 파일/디렉토리가 부모 디렉토리의 그룹을 상속하기 쉽습니다

협업 디렉토리에서 자주 사용합니다.

---

### 12-3. ACL은 왜 쓰나?

기본 권한만으로도 어느 정도는 되지만, 더 세밀하게 제어하고 싶을 때 ACL을 씁니다.

특히:

- 여러 그룹이 얽힐 때
- 새 파일에도 권한 정책을 유지하고 싶을 때
- 협업 디렉토리 정책을 안정화하고 싶을 때

유용합니다.

확인 명령:

```bash
getfacl /path/to/dir
```

## 13. 꼭 알아야 하는 Bash 기초 문법

사용자가 `scripts`를 거의 모른다고 했으니, 정말 기초부터 짚겠습니다.

---

### 13-1. 실행 권한

스크립트 파일은 텍스트 파일일 뿐입니다.

실행 가능하게 하려면 보통:

```bash
chmod +x monitor.sh
```

또는 요구사항대로:

```bash
chmod 750 monitor.sh
```

처럼 실행 비트를 줘야 합니다.

---

### 13-2. 현재 셸에서 변수 읽기

예:

```bash
echo "$AGENT_HOME"
```

이건 변수 값을 출력합니다.

따옴표를 쓰는 이유:

- 공백이나 특수문자가 있어도 안전하게 한 덩어리로 다루기 위해서입니다.

---

### 13-3. 단일 따옴표와 이중 따옴표

```bash
'$AGENT_HOME'
"$AGENT_HOME"
```

차이:

- 작은따옴표 `'` 안에서는 변수 치환이 안 됨
- 큰따옴표 `"` 안에서는 변수 치환이 됨

예:

```bash
echo '$AGENT_HOME'
echo "$AGENT_HOME"
```

결과:

- 첫 번째는 글자 그대로 `$AGENT_HOME`
- 두 번째는 실제 값

---

### 13-4. `&&` 와 `||`

`&&`:

> 앞 명령이 성공하면 뒤 명령 실행

`||`:

> 앞 명령이 실패하면 뒤 명령 실행

예:

```bash
mkdir test && cd test
```

디렉토리 생성이 성공했을 때만 이동합니다.

예:

```bash
stat file.txt || echo "missing"
```

`stat`가 실패하면 `missing`을 출력합니다.

---

### 13-5. 종료 코드 확인

직전 명령의 종료 코드는:

```bash
echo $?
```

로 볼 수 있습니다.

이 값이 `0`이면 성공, 아니면 실패입니다.

## 16. 체크리스트 답변 연습용 모범 문장

여기서는 진짜로 말로 설명할 때 쓸 수 있게 짧고 단단한 문장을 적어 둡니다.

### Q. 왜 SSH 포트를 바꿨나요?

A. 기본 22번 포트는 자동화 스캔과 무차별 대입 시도가 매우 많아서, 포트를 변경해 기본 노출면을 줄였습니다. 강한 인증을 대체하는 것은 아니지만 기본적인 하드닝 조치입니다.

### Q. 왜 Root 원격 로그인을 막았나요?

A. Root는 최고 권한 계정이므로 직접 원격 노출되면 위험이 큽니다. 일반 계정으로 접속한 뒤 필요한 순간에만 `sudo`를 쓰게 해서 권한 남용 위험과 감사 추적 문제를 줄였습니다.

### Q. 왜 `agent-common`과 `agent-core`를 나눴나요?

A. 공유가 필요한 영역과 민감한 영역을 분리하기 위해서입니다. 업로드 디렉토리는 공통 그룹이 쓰고, 키와 로그 같은 민감 자원은 core 그룹만 접근하게 해서 최소 권한 원칙을 적용했습니다.

### Q. 왜 프로세스와 포트를 둘 다 확인하나요?

A. 프로세스가 살아 있어도 포트를 열지 못하면 서비스는 정상 제공 상태가 아닐 수 있습니다. 그래서 생존 여부와 실제 서비스 가능 여부를 분리해서 둘 다 확인했습니다.

### Q. 왜 로그 형식을 고정했나요?

A. 사람이 읽기 쉽고, 후속 스크립트가 안정적으로 파싱해 통계를 낼 수 있도록 하기 위해서입니다.

### Q. 왜 `>>`를 썼나요?

A. 모니터링 로그는 과거 이력이 계속 쌓여야 하므로 덮어쓰기 `>`가 아니라 추가 `>>`를 사용했습니다.

### Q. 왜 경고와 실패를 구분했나요?

A. 프로세스/포트처럼 서비스 생존과 직접 연결된 항목은 실패로 처리하고, 방화벽 비활성이나 임계값 초과는 즉시 서비스 중단을 뜻하지 않으므로 경고로 남겨 운영자가 후속 대응할 수 있게 했습니다.

## 18. 아주 짧은 최종 정리

이 미션의 본질은 다음 한 문장으로 요약할 수 있습니다.

> 보안이 적용된 리눅스 서버에서, 역할 기반 권한 체계를 만들고, 제공 앱을 올바른 환경으로 실행한 뒤, Bash 스크립트와 cron으로 상태를 지속적으로 감시하고 기록하는 훈련

그리고 `scripts/`는 단순한 "정답 파일 모음"이 아니라 아래를 배우기 위한 재료입니다.

- Bash 문법
- 리눅스 운영 명령어
- 권한 설계
- 로그 설계
- 운영 자동화 사고방식

이 문서를 읽고 나면 최소한 아래 정도는 자연스럽게 말할 수 있어야 합니다.

- 왜 이 설정을 했는지
- 어떤 명령어로 검증하는지
- 스크립트가 어떤 순서로 동작하는지
- 문제가 생기면 어디부터 볼지

그 상태가 되면 체크리스트를 "외워서 통과"하는 게 아니라, 이해하고 통과하게 됩니다.
